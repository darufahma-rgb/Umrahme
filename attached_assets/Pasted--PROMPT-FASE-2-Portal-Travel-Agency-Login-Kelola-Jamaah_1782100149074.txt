# PROMPT FASE 2 — Portal Travel Agency (Login + Kelola Jamaah Sendiri)

> Prasyarat: Fase 1 (tabel `jamaah_accounts`, verifikasi nama saat login jamaah) sudah selesai dan berjalan.
>
> Tujuan: travel agency bisa login sendiri ke portal mereka, dan hanya bisa melihat/mengelola jamaah di bawah tenant mereka sendiri — tidak bisa mengakses data tenant lain.

## 1. Tabel mapping `tenant_users` (link akun auth ke tenant)

```sql
create table tenant_users (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  tenant_id uuid not null references tenants(id) on delete cascade,
  created_at timestamptz not null default now(),
  unique (user_id, tenant_id)
);

create index idx_tenant_users_user on tenant_users (user_id);
create index idx_tenant_users_tenant on tenant_users (tenant_id);
```

Tabel ini yang menentukan: akun Supabase Auth dengan `user_id` ini punya akses ke `tenant_id` mana. Super admin (lo) bisa assign satu akun ke satu tenant. Satu user bisa punya akses ke beberapa tenant jika diperlukan nanti, tapi untuk MVP fokus 1 user = 1 tenant.

### RLS untuk `tenant_users`
- AKTIFKAN RLS.
- Policy READ: `using (user_id = auth.uid())` — user hanya bisa lihat mapping milik dirinya sendiri.
- Policy WRITE: TIDAK ada policy publik.

### Update RLS `jamaah_accounts` untuk travel agency
Tambahkan policy baru (di samping policy publik read yang sudah ada dari Fase 1):
```sql
-- Travel agency bisa INSERT/UPDATE/DELETE jamaah di tenant mereka
create policy "travel_agency_write_jamaah" on jamaah_accounts
  for all
  using (
    exists (
      select 1 from tenant_users
      where tenant_users.user_id = auth.uid()
      and tenant_users.tenant_id = jamaah_accounts.tenant_id
    )
  );
```

## 2. Cara setup akun travel agency (dilakukan oleh super admin/lo)

Super admin membuat akun travel agency via Supabase Dashboard (Authentication → Users → Add user), lalu insert ke `tenant_users`:
```sql
insert into tenant_users (user_id, tenant_id)
values ('<uuid-user-travel-agency>', '<uuid-tenant-mereka>');
```

Dokumentasikan ini di `src/config/tenants/README.md` (append, jangan hapus isi lama) sebagai panduan langkah menambah akun travel agency baru.

## YANG TIDAK BOLEH BERUBAH

- Admin panel (`/admin/*`) dan `AdminAuthContext` — TIDAK disentuh sama sekali. Super admin tetap bisa akses semua tenant dari panel yang sama.
- App jamaah (`/login`, `/beranda`, dll) — TIDAK disentuh.
- Tabel `jamaah_accounts` dan `tenants` yang sudah ada — hanya ditambahkan RLS policy baru, tidak ada perubahan kolom.
- Tipe `TenantRow`, `JamaahAccountRow`, dll yang sudah ada di `supabase.ts` — tidak diubah, hanya ditambah `TenantUserRow`.

## CATATAN PENTING: isolasi data

Travel agency HANYA boleh lihat/edit jamaah di tenant mereka sendiri. Ini dijamin di dua lapis:
1. **RLS di database** (poin 1) — bahkan kalau ada bug di frontend, database tetap menolak akses lintas-tenant
2. **Query di frontend** selalu filter by `tenant_id` yang didapat dari `TravelAuthContext` (bukan dari input user) — jangan pernah trust `tenant_id` dari URL parameter atau localStorage untuk operasi write
